# 隔离机制设计

## 为什么需要隔离

多 Agent 协作时，如果所有 Agent 都能直接修改论文文件，会导致：
- 冲突：多个 Agent 同时修改同一个文件
- 质量失控：子 Agent 产出未经审阅就写入论文
- 安全风险：Agent 可能误操作删除或覆盖重要内容

隔离机制从多个层面解决这些问题。

## 角色权限

系统定义了三种角色，权限递进：

| 角色 | 读 core | 写 core | 写 overlay | Git 操作 |
|------|---------|---------|------------|----------|
| Assistant（主 Agent） | ✓ | ✓ | ✓ | ✗ |
| Worker（子 Agent） | ✓ | ✗ | ✓（仅自己的） | ✗ |
| GitAgent | ✓ | ✓ | ✓ | ✓ |

角色定义在 `config/roles.json` 中。

## Copy-on-Init Overlay 模型

Worker 创建时，系统将 core 中的所有项目文件浅拷贝到 overlay 目录（`init_overlay()`）。之后 Worker 的所有操作（读、写、bash、编译）都在 overlay 内完成，不再 fallback 到 core。

```
创建 Worker Session
    ↓
init_overlay(): core 文件 → overlay（跳过 .git、.bot、__pycache__ 等）
    ↓
Worker 在 overlay 内工作（resolve() 直接返回 overlay 路径）
    ↓
完成后 _diff_overlay(): SHA-256 比较，找出变更文件
    ↓
只合并变更文件到目标目录
```

### 跳过拷贝的目录（`_SKIP_COPY`）

`.bot`、`.git`、`__pycache__`、`.DS_Store`、`subagents`、`task_workers`、`_task_workers`、`_subagent_results`，以及所有以 `.` 开头的路径。

### 为什么用 Copy-on-Init 而不是透明 Overlay

旧模型（读时 fallback core，写时落 overlay）存在以下问题：
- `latex_compile` 直接走 `project.core`，看不到 Worker 的修改
- `bash ls` 和 `ls` 工具行为不一致（bash 走 `resolve(".")`，ls 走合并逻辑）
- 每个工具都需要理解双层查找语义，容易出 bug

Copy-on-Init 后，overlay 是自包含的完整工作目录，所有工具行为一致。

## Merge 流程

Worker 完成任务后，通过 `_diff_overlay()` 找出相对 core 有变更的文件（SHA-256 比较），只合并这些文件：

- **SDD Worker**：变更文件合并到 `core/_task_workers/{task_id}_r{round_id}/`
- **Swarm Subagent**：根据 `merge_to_core` 配置合并到 core 或 `core/_subagent_results/{agent_name}/`

未修改的 core 文件不会被合并，避免冗余拷贝。

## 路径解析

`Session.resolve()` 根据角色返回不同路径：

- **Worker**：直接返回 `overlay / path`（overlay 已有完整副本）
- **Assistant**：直接返回 `core / path`（与 `Project.resolve()` 一致）

`Session.write_target()` 同理：Worker 写 overlay，Assistant 写 core。

## 路径安全

所有文件操作都经过路径校验：

- `..` 被拒绝：防止通过相对路径逃出工作目录
- 绝对路径被拒绝：所有路径必须是相对于 core 的
- Bash 命令的工作目录锁定在 overlay（Worker）或 core（Assistant）

## 写入保护

- 截断的写入操作会被检测并拒绝，防止文件被部分覆盖导致损坏
- 超过 32KB 的工具输出会被自动截断（写入工具除外）
- 每次写入都被追踪到 pending_writes 列表，turn 结束时统一提交

## 依赖目录写保护（Task Mode）

Task Mode 中，Worker 会收到上游任务的产出作为依赖。系统通过三层防御阻止 Worker 修改依赖：

| 层级 | 机制 | 拦截范围 |
|------|------|----------|
| Layer 1 | `Session._protected_prefixes` 检查 | write_file、str_replace |
| Layer 2 | Worker prompt 中注入 READ-ONLY 警告 | LLM 行为引导 |
| Layer 3 | OS chmod 444/555 | bash echo、bash cp 等 |

bash cp 从只读文件复制时，目标文件会继承 444 权限。`Session.write_target()` 检测到目标不可写时自动恢复 644。

详见 `README/design/17_Task_Mode.md`。

## 关键设计决策

**为什么用 Copy-on-Init？** 统一语义。Worker 的 overlay 是完整的工作目录，所有工具（bash、read_file、latex_compile）行为一致，不需要理解双层查找。

**为什么默认 merge_to_core = false？** 安全优先。让主 Agent 有机会审阅子 Agent 的产出，避免低质量内容直接进入论文。

**为什么 GitAgent 是独立角色？** Git 操作（特别是回退、丢弃）是破坏性的，需要专门的权限控制和确认机制，不应该混在普通的文件操作权限中。

- `config/roles.json` — 角色定义
